# Что значит CI

**Continuous integration - Непрерывная интеграция служит для создания пользовательских рабочих процессов непрерывной интеграции (CI) непосредственно репозитории GitHub с помощью GitHub Actions.**


Непрерывная интеграция (CI) - это практика разработки программного обеспечения, которая требует частых коммитов в общем репозитории. Фиксация кода чаще всего позволяет быстрее обнаружить ошибки и сокращает объем кода, который разработчик должен отлаживать при поиске источника ошибки. Частые обновления кода также упрощают объединение изменений, внесенных разными членами команды разработчиков программного обеспечения. Это удобно для разработчиков, которые могут тратить больше времени на написание кода и меньше - на отладку ошибок или разрешение конфликтов при слиянии.


Для создания и тестирования вашего кода требуется сервер. Можно создавать и тестировать обновления локально, прежде чем отправлять код в репозиторий (у нас этим занимается husky, который запускает команду   "eslint . && tsc --noEmit", которая проверяет:
  - Код соответствует правилам линтера,
  - В проекте нет TypeScript-ошибок,
  - Ничего не сломается при сборке).

Или вы можем использовать сервер CI, который проверяет наличие новых коммитов кода в репозитории.

*Husky и CI могут проверять одни и те же вещи, но они не дублируют друг друга полностью*.

### Чем отличаются husky от CI:

1. **Husky — локальная проверка перед коммитом**
  - Работает на вашей машине при попытке закоммитить код (pre-commit hook).
  - Запускает команды, которые вы настроили, например:

eslint . && tsc --noEmit

Что делает:
  - Проверяет, что код соответствует правилам линтера
  - Проверяет типы TypeScript
  - Блокирует коммит, если есть ошибки

*Цель: предотвращает попадание некорректного кода в репозиторий.*

**Плюс: вы получаете мгновенный feedback локально, до того как пушить изменения.**

2. **CI (GitHub Actions) — серверная проверка, работает в удалённом репозитории (GitHub), обычно при пуше или pull request**
  - Может запускать те же проверки (eslint, tsc), а также:
  - сборку проекта (npm run build)
  - запуск тестов (npm test)
  - деплой, lint + форматирование, проверку зависимостей

*Цель: гарантировать, что любой код в ветке (например, main) проходит стандарты проекта и сборку на сервере.*

**Плюс: проверки выполняются в единой среде, одинаковой для всех участников команды.**

3. Дублируют ли они друг друга - частично — оба запускают eslint и tsc.

**Отличие: Husky работает локально перед коммитом, CI — на сервере после пуша.**

Почему оба нужны:
  - Husky помогает не допустить ошибок в коммитах
  - CI помогает гарантировать, что всё работает в ветке, даже если кто-то обходит Husky (например, через git commit --no-verify)

### CI-тесты

GitHub запускает CI-тесты и предоставляет результаты каждого теста в pull-запросе, чтобы вы могли увидеть, приводит ли изменение в вашей ветке к ошибке.

Когда все CI-тесты в рабочем процессе проходят успешно, внесенные вами изменения готовы для проверки членом команды или объединения. Если тест завершается неудачей, возможно, причиной сбоя стало одно из внесенных вами изменений.

**Правила прописанные в .github/workflows/ci.yml**

ci.yml - инструкция для GitHub когда кто-то делает push или открывает Pull Request, GitHub автоматически:
  - устанавливает зависимости
  - запускает линтер
  - проверяет типы
  - собирает проект

**Если что-то ломается — вы сразу это видите.**


### Где находится файл ci.yml?

Он должен лежать строго здесь:
```.github/workflows/ci.yml
```

Структура папок:

```project/
 ├─ src/
 ├─ package.json
 ├─ vite.config.ts
 └─ .github/
     └─ workflows/
         └─ ci.yml
```
**Если файл лежит не в этой папке — GitHub его не увидит.**

ci.yml - не локальный файл. Он работает на серверах GitHub.
**Создаём один раз — дальше GitHub всё делает сам.**

### Когда запускается:
  - при push в main
  - при создании PR

### Что происходит:
  - GitHub создаёт виртуальный Linux-сервер.
    ```steps:
      - uses: actions/checkout@v4
```
  - Скачивает ваш репозиторий на сервер.
      ```- uses: actions/setup-node@v4
        with:
          node-version: 20
```
  - Устанавливает Node.js.
      ```- run: npm install
```
  - Устанавливает зависимости.
      ```- run: npm run check
      - run: npm run build
```
  - Запускает скрипты:
    - ESLint
    - TypeScript
    - сборку

### Что происходит после этого?

Если всё прошло успешно — вы видите зелёную галочку ✅
Если что-то сломалось — красный крест ❌

Вкладка находится здесь:

```GitHub → репозиторий → Actions
```

### Редактирование ci.yml

Если вы планируете редактировать ci.yml часто стоит установить расширение 'GitHub Actions' extensions from GitHub for ci.yml.
Для GitHub Actions нужен специальный плагин:
  - Имя: GitHub Actions
  - Разработчик: GitHub

Функции:
  - Подсветка синтаксиса .yml для Actions
  - Автодополнение для jobs, steps, uses, run
  - Проверка базовых ошибок формата
  - Навигация по Actions документации прямо в редакторе

**VSCode не отправляет ничего на GitHub сам по себе. Расширение — только для удобства локально.**


### Что такое workflow в GitHub Actions?

**Workflow — это автоматический сценарий, который GitHub запускает при определённом событии.**

*Когда происходит событие → GitHub запускает виртуальный компьютер → выполняет команды → показывает результат.*

Этот виртуальный компьютер существует 1–2 минуты, выполняет задачи и исчезает.

Как это реально работает:

1. делаем `git push origin develop`

2. GitHub получает push, проверяет есть ли .github/workflows/*.yml?
Если есть — запускает workflow:
  - Создаёт временный сервер (ubuntu).
  - Клонирует ваш репозиторий.
  - Устанавливает Node.
  - Делает npm install.
  - Делает npm run check.
  - Делает npm run build.

Если всё ок → зелёная галочка. Если ошибка → красный крест.


### CI решает реальные проблемы.

1. Проверка кода до merge
Представим, студент забыл запустить линтер, другой случайно сломал сборку, третий добавил код, который не компилируется. Локально у него может работать, но у другого разработчика — нет.

*CI говорит: "Проект не собирается. Исправь перед merge."*

2. Проверка Pull Request
Если кто-то делает PR в develop CI автоматически запускается, сразу видим  Build passed или failed.
И не нужно проверять руками.

3. Гарантия, что проект вообще собирается
Иногда у одного работает, у проверяющего — нет.

**CI запускается в чистой среде.**
Если сборка проходит там — она воспроизводима.

4. Это стандарт индустрии

Любой реальный проект использует CI:
  - проверка линтера
  - проверка типов
  - тесты
  - сборка
  - деплой
